How it's built

§ ii · technical notes
01 · Widget tree
Hierarchical, event-propagating
Every UI piece (button, slider, canvas, container) is a widget. Widgets nest inside other widgets. Mouse and keyboard events walk down the tree, and each widget decides whether to handle the event or pass it to its children.
02 · Drawing model
Immutable graphics context
Instead of mutating a canvas, each draw command returns a new context with the stroke added. Makes undo/redo trivial: you just swap the old context back in. Pure functional, no hidden state.
03 · Undo stack
Deque-backed history
A double-ended queue holds every drawing state. Pressing u pops the latest frame off and re-renders. Memory-bounded so the history doesn't grow forever. Two-way, so redo works too.
04 · Runtime
Custom event loop
OCaml doesn't ship a browser runtime, so there's a hand-written loop that polls the DOM via js_of_ocaml bindings, dispatches mouse/keyboard events into the widget tree, and repaints. ~100 lines of glue.
